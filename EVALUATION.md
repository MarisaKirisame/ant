# Artifact Evaluation Guide

This guide maps the claims in *Incremental Live Programming via Shortcut
Memoization* to the reports generated by the artifact. Run commands from the
top-level artifact directory.

The artifact uses Docker as its reproducible environment. The image is built
with network access so it can download pinned dependencies; all experiment and
proof containers run with Docker networking disabled.

## Scope and Requirements

The artifact supports:

- the Agda mechanization corresponding to Section 2;
- the Chordata implementation described in Sections 3 and 4; and
- every empirical experiment in Section 5, including the official-Hazel
  comparison and cache-eviction ablation.

The three anonymized trace contributors are named `user1`, `user2`, and
`user3`. The older initials in the archived paper results refer to these same
three trace groups.

Use a 64-bit Linux host with Docker Engine and at least 32 GiB of free disk
space. The paper's reference experiments ran on Ubuntu 24.04.4, an Intel
i7-8700K, and 32 GiB of memory. Exact performance values vary with hardware and
system load; compare paired configurations from the same run.

The account running the evaluation must have permission to access the Docker
daemon without `sudo`. If Docker was just installed or the account was just
added to the `docker` group, log out and back in before starting so the new
group membership takes effect. A Docker permission error occurs before any
artifact build or proof checking begins.

## Evaluator Interface

The evaluator needs only these four commands:

| Command | Purpose |
|---|---|
| `make smoke` | Kick-the-tires check of the complete experimental path |
| `make quick` | Focused 22-trace Hazel evaluation at input size 400 |
| `make eval` | Complete evaluation and all empirical reports |
| `make proof` | Complete Agda proof check |

Each of the three evaluation commands (`make smoke`, `make quick`, and
`make eval`) deletes its intermediate results and generated reports before it
starts. The resulting report therefore contains only measurements produced by
that command.

The artifact contains no pre-generated measurements or reports from the
authors' machine. The final-paper values listed below are comparison
checkpoints, not bundled result files or machine-independent acceptance
thresholds.

The first invocation builds the pinned Docker image. This installs OCaml 5.4.1
for Chordata, OCaml 5.2.0 for Hazel, Agda 2.7.0.1 with standard library 2.2,
and the locked reporting dependencies.

## Smoke

`make smoke` runs reduced instances of every experiment family and exercises
the complete build, measurement, plotting, and reporting path. It uses input
size 2, one Hazel candidate per trace, one arithmetic sample, and scaling size
2.

Success means the command exits with status zero and produces the report index
at `docker-output/output/index.html`. Smoke values are structural checks and
must not be compared with the paper's numbers.

## Quick

`make quick` runs the existing Hazel-only evaluation: all 22 terminating
participant/benchmark combinations, every recorded program state, and input
size 400. It takes approximately 15 minutes on the reference machine.

Read `docker-output/output/hazel/index.html`. The expected qualitative result
is that memoized Chordata is substantially faster than the non-memoizing CEK
over the complete trace set.

Quick deliberately omits the official Hazel evaluator, eviction ablation,
arithmetic benchmark, scaling study, and input-regularity study. It provides a
meaningful focused result, but it is not the complete paper reproduction.

## Full Evaluation

`make eval` runs all experiments and writes the report index to:

`docker-output/output/index.html`

The complete configuration includes:

- all 22 terminating Hazel participant/benchmark combinations at input size
  400;
- official Hazel and the Chordata CEK baseline at input size 10;
- a per-mode official-Hazel timeout of 10 minutes;
- eviction and no-eviction configurations;
- arithmetic evaluation;
- Hazel and arithmetic scaling sizes 10, 20, 40, 100, 200, and 400; and
- input-regularity sizes 10, 20, 40, 100, 200, 400, 1000, 2000, and 4000.

The two omitted combinations, User 3 merge sort and pair, contain infinite
loops. They appear as `X` in the paper rather than receiving performance
factors.

Allow approximately three hours for the evaluation; slower hosts may take
longer. The external Hazel evaluator may time out or exhaust memory on
difficult sorting traces. For example, `user2_qs` reached the 10-minute limit
on the authors' test machine. These outcomes are recorded and do not abort the
pipeline; official-Hazel aggregates include only successful rows.

## Reading the Full-Evaluation Report

All empirical claim checking begins at
`docker-output/output/index.html`, generated by `make eval`. The index links
the following pages:

| Paper claim | Open from the report index | What to inspect |
|---|---|---|
| RQ1: reuse across program changes | **Hazel Report** | Aggregate memoized-versus-CEK speedup, per-user/benchmark results, scatterplot, CDF, component costs, and peak-memory overhead |
| Official Hazel comparison | **Hazel Report** → **CEK vs Hazel Baseline** | Successful official-Hazel/CEK paired timings and timeout or out-of-memory statuses |
| Cache-eviction ablation | **Hazel No-Eviction Ablation** → **Eviction Ablation** | Paired execution-time and peak-memory ratios with and without eviction |
| RQ2: repetitive computation | **Arith Report** | Arithmetic memoized-versus-CEK scatterplot and geometric speedup |
| RQ3: scaling | **Scaling Report** → Hazel and arithmetic pages | Speedup-versus-input-size plots |
| Input regularity | **Scaling Report** → input-regularity page | Speedup by Random, Block, Change1, and Constant input patterns |

### RQ1: Reuse Across Program Changes

The Hazel report supports the abstract, introduction, Section 5.1, and
conclusion claims. Read its aggregate summary, per-benchmark pages, scatterplot,
and CDF.

Expected qualitative result: memoized execution is substantially faster than
CEK over the complete trace set, and most program-state points lie below the
equal-time line. Memory is summarized per trace: the maximum positive
resting-adjusted memo live-word measurement is divided by the corresponding
CEK maximum, then trace-level ratios are aggregated geometrically.

The nested official-Hazel page compares evaluation time only, deliberately
excluding Hazel parsing and elaboration.

### Cache-Eviction Ablation

Read the eviction-ablation page linked from the no-eviction report.

Expected qualitative result: eviction has a small execution-time cost and
reduces peak memory. Time measurements are aligned per execution. Memory uses
one independently measured peak per participant/benchmark/configuration before
the ratios are aggregated.

### RQ2: Reuse Within Repetitive Computation

Read the arithmetic report and its scatterplot.

Expected qualitative result: memoized Chordata is faster geometrically despite
having no reuse between experimental points. Expressions have size 400, and
the cache is cleared between points.

### RQ3: Scaling With Input Size

Read the Hazel and arithmetic pages linked from the scaling report.

Expected qualitative result: speedup grows with input size approximately
linearly on the log-log plots. Hazel can exceed equal time at small inputs
because editing traces reuse work across changes; arithmetic requires larger
inputs before within-execution repetition outweighs memoization overhead.

### Input Regularity

Read the input-regularity page linked from the scaling report. Artifact labels
map to the paper as follows:

| Artifact identifier | Paper label |
|---|---|
| `random` | Random |
| `block` | Block |
| `mod1` | Change1 |
| `same` | Constant |

Expected qualitative result:

- Random speedup remains approximately flat as size grows;
- Constant and Change1 speedups increase with size;
- Block lies between those cases; and
- Block and Change1 commonly form separate clusters for linear list functions
  and sorting functions.

## Paper Table Mapping

Table 1 in the paper, “Detailed results for the live-programming benchmark,”
is assembled from two table-valued macros generated in
`output/hazel/hazel_result.tex`:

| Paper table | Contents | Generated macro |
|---|---|---|
| Table 1a, “Speedup and memory overhead by benchmark” (`tab:individual`) | Time speedup and peak-memory overhead for each benchmark and user | `\hazelSpeedupTable` |
| Table 1b, “Time breakdown by component” (`tab:component`) | Percentage of memoized execution time spent looking up, composing, instantiating, applying, and inserting rules | `\hazelSpeedBreakdown` |

After `make eval`, both macros are in
`docker-output/output/hazel/hazel_result.tex`. The paper imports that generated
file and places the two macros inside the Table 1 subtables. The scalar macros
in the reference table below supply values used in the paper's surrounding
text; they do not expand to complete tables.

## Reference Checkpoints

The table below shows the results reported in the final paper. A new
evaluation should reproduce the same overall trends, but exact values may
differ because performance measurements depend on the machine and system load.

| Quantity | Final-paper value | Generated LaTeX result |
|---|---:|---|
| Recorded Hazel programs in timing aggregates | 4,615 | `output/hazel/hazel_result.tex`: `\hazelPointCount` |
| Memoized Chordata / CEK geometric speedup | 30.7x | `output/hazel/hazel_result.tex`: `\hazelTotalSpeedup` |
| Memoized Chordata / CEK aggregate memory overhead | 7.131x | `output/hazel/hazel_result.tex`: `\hazelTotalMemoryOverhead` |
| Official Hazel / Chordata CEK geometric time ratio | 8.232x | `output/hazel/hazel_result.tex`: `\hazelBaselineGeoMean` |
| Evicting / no-eviction geometric time ratio | 1.048x | `output/hazel-no-evict/hazel_no_evict_result.tex`: `\hazelNoEvictTotalSpeedup` |
| No-eviction / evicting aggregate memory ratio | 1.212x | `output/hazel-no-evict/hazel_no_evict_result.tex`: `\hazelNoEvictTotalMemoryOverhead` |
| Arithmetic memoized / CEK geometric speedup | 2.673x | `output/arith/arith_result.tex`: `\arithTotalSpeedup` |

These paths are relative to `docker-output/`, so, for example, the generated
Hazel file is `docker-output/output/hazel/hazel_result.tex`. The same values
are displayed in the linked HTML reports.

The eviction-ratio directions are intentional: `1.048x` means eviction takes
1.048 times the no-eviction time, while `1.212x` means no-eviction uses 1.212
times the evicting configuration's peak memory.

## Proof

`make proof` type-checks `proof/all.agda`, which imports:

| Paper item | Module |
|---|---|
| Definitions 2.1–2.5 and 2.8 | `proof/core.agda` |
| Theorem 2.6, Compatibility of Rule Composition | `proof/compatibility.agda` |
| Theorem 2.7, Shortcut Existence | `proof/existence.agda` |
| Theorem 2.9, Shortcut Validity | `proof/validity.agda` |

The trusted assumptions are function extensionality, a generic constructor
set, and an infinite variable set with the operations and decidable equality
declared in `proof/core.agda`.

## Interpreting Deviations

- Timing depends on CPU speed, system load, virtualization, and frequency
  scaling. Evaluate ratios and trends rather than identical nanosecond values.
- Live-word measurements depend on the OCaml runtime and should be compared
  only between paired modes in this pinned image.
- A participant/benchmark cell can vary more than the aggregate because trace
  lengths differ and some traces contain a small number of expensive states.
- `X` for User 3 merge sort and pair is expected because those traces contain
  infinite loops.
- Official-Hazel modes that time out or exhaust memory appear explicitly in
  the per-mode status table and are excluded from aggregate statistics.
